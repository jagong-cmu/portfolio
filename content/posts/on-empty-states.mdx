The empty state is the screen a user sees before they've done anything: no
projects yet, no messages, no history. It's easy to skip, because when you're
building you always have test data sitting in front of you. You rarely look at
the version with nothing in it.

But the empty state is the **first** thing a new user sees. It's not an edge
case — it's the front door.

## Three jobs an empty state does

1. **Explains** what this screen is for, in one line.
2. **Reassures** that nothing is broken — emptiness is expected.
3. **Invites** the single most useful first action.

When all three are missing, a new user reads emptiness as failure. They assume
the page didn't load, or that they did something wrong.

## Write the empty state first

The trick that changed how I build: design the empty state *before* the full
one. If you can make the zero-data screen feel intentional and welcoming, the
populated screen almost designs itself.
